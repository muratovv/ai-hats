# Glossary

Core concepts used across ai-hats code and docs. One short definition per term, plus a pointer to the canonical detail. **Not** a catalog ŌĆö for the full listing of roles, skills, traits, or CLI commands use `ai-hats list ...` and `ai-hats --tree`.

This file is the naming source-of-truth. When another doc needs to define a core term, link here instead of redefining it.

---

## Provider

A target LLM CLI that ai-hats wraps: `claude` or `gemini`. The choice lives in `ai-hats.yaml` (`provider:`). One role composition produces two injection targets ŌĆö `CLAUDE.md` or `GEMINI.md` ŌĆö built during `ai-hats self init` / `self update` (the `self bump` CLI verb was removed in HATS-470; bump runs implicitly inside those flows via `ai_hats._bump_internal`). Switching keeps composition intact: `ai-hats config set -p <provider>`.

Detail ŌĆö see [1].

## Session

One invocation of the provider CLI under a chosen role. Entry points: `ai-hats` (no subcommand), `ai-hats agent <role>`, `ai-hats execute ŌĆ”`. Per-session artefacts land in `<ai_hats_dir>/sessions/runs/session_<id>/` (`audit.md`, `metrics.json`, `usage.json` ŌĆö the machine-readable `usage/v1` context-cost report, HATS-664; `transcript.txt`, `meta_prompt.txt` ŌĆö the exact bytes the provider saw as system-prompt override, HATS-523). The runtime ends a session with a `session_end` event that may trigger a per-session retrospective ŌĆö see [Reflect](#reflect).

Lifecycle diagram ŌĆö see [2].

**Parent escape-hatch** ŌĆö a deterministic way out of a *wedged* interactive (HITL) session. The provider runs under a PTY proxy whose stdin is in raw mode, so Ctrl-C is forwarded to the child as a byte; if the child stops honouring it and never exits, the user is otherwise trapped until an external kill. The runtime counts consecutive Ctrl-C (`0x03`) within a ~1.5 s sliding window: the first two are forwarded unchanged (dormant for healthy sessions, which exit on the second Ctrl-C via master EOF), and the third is intercepted ŌĆö the parent runs its bounded shutdown and returns exit code **130** (so a wedged session is no longer mis-reported as success). Source: `runtime._scan_escape` + `WrapRunner._pty_spawn` (HATS-679).

## Role

A root composition that the agent wears during a session ŌĆö bundles traits, rules, skills, and an injection block into one config. The shipped library splits into two layers: `library/core/roles/` (engine-internal: `initial-wizard`, `session-reviewer`, `judge-auditor`, `judge`, `judge-for-role`, `auditor-for-role`, `hypothesis-intake`, `test-agent`) and `library/usage/roles/` (curated user-facing: `assistant`, `dev-python`, `dev-web`, `maintainer`, `architect`, `sre`, `go-dev`, `go-dev-full`). Catalog ŌĆö `ai-hats list roles`; layered structure and override precedence ŌĆö see [9]. Example: [`library/usage/roles/assistant/config.yaml`](../library/usage/roles/assistant/config.yaml). Customization (add / remove / override) ŌĆö see [6].

Key system roles you will meet in cross-doc prose:

- `initial-wizard` ŌĆö interactive setup that runs on `ai-hats self init`. See [6].
- `session-reviewer` ŌĆö per-session retrospective; votes on active HYPs and files a PROP on self-problem. Triggered by `ai-hats reflect session` (auto on `session_end` per policy, or manual). See [5].
- `judge-auditor` / `judge` / `judge-for-role` / `auditor-for-role` ŌĆö the reflection-loop roles. Backlog triage runs two-phase: `judge-auditor` (Phase 1, headless, read-only audit) ŌåÆ `judge` (Phase 2, HITL, ack'd mutations) via `ai-hats reflect hypothesis`. Role-coherence audits use `auditor-for-role` ŌåÆ `judge-for-role` via `ai-hats reflect role`. See [5].

## Trait

An ai-hats-native composition primitive: a reusable bundle (rules + skills + injection text) included by one or more roles. Traits are the unit of cross-role reuse ŌĆö a fix in one trait reaches every role that pulls it in on the next `ai-hats self init`. Flat model: a trait cannot include another trait. Format: `library/{core,usage}/traits/<name>/config.yaml`. Catalog ŌĆö `ai-hats list traits`. Composition rules ŌĆö see [3]; library layout ŌĆö see [9].

Key system traits every role inherits transitively:

- `trait-base` ŌĆö minimum behaviour for every role: core principles (safety > integrity > convenience > velocity), pessimistic verification, brevity, least astonishment.
- `trait-agent` ŌĆö agent-mode loop primitives: backlog state machine, delegation pattern, memory hygiene (context-reset / handoff / summary), anti-anchoring, tool-call hygiene.

## Rule, Skill

The two component kinds that ai-hats injects into the **provider** prompt (`CLAUDE.md` / `GEMINI.md`). They apply at the provider layer ŌĆö the LLM reads them and follows; ai-hats does not interpret their content.

| Component | What it is                                                  | Format (under `library/{core,usage}/ŌĆ”`)                                 |
| --------- | ----------------------------------------------------------- | ----------------------------------------------------------------------- |
| **Rule**  | Behavioural constraint (do / don't). No decision logic.     | `rules/<name>/rule.md` + `metadata.yaml`                                |
| **Skill** | Procedure, checklist, or protocol with steps and branching. | `skills/<name>/SKILL.md` (+ `metadata.yaml`, `scripts/`, `references/`) |

Catalog ŌĆö `ai-hats list {rules,skills}`. Formats ŌĆö see [3]; library layout and override precedence ŌĆö see [9].

**Rule-delivery contract (HATS-700).** Not every composed rule's body reaches the prompt. Only the **always-on** set (`providers.ALWAYS_ON_RULES` ŌĆö safety-critical + framework invariants) is injected in full under the prompt's `## RULES` section; every other composed rule is **provenance only** ŌĆö its body is not delivered, and its canonical guidance is the compact summary in the trait/role injection that references it. A `see rule X` pointer must therefore name a rule that is always-on or registered in `SUMMARIZED_IN_INJECTION` (`ai_hats/rule_delivery.py`); the **rule-delivery-gate** pre-commit hook and the G2 unit test (`find_dangling_rule_pointers`) enforce this so an author never ships a pointer to a rule the agent cannot read.

## Backlog

Three kinds of cards with strict state machines. **All operations go through the `ai-hats task ŌĆ”` CLI** ŌĆö direct access to `<ai_hats_dir>/tracker/**` is forbidden.

| Kind                 | ID         | Lifecycle                                                                                          |
| -------------------- | ---------- | -------------------------------------------------------------------------------------------------- |
| **Task**             | `HATS-NNN` | `brainstorm ŌåÆ plan ŌåÆ execute ŌåÆ document ŌåÆ review ŌåÆ done` (plus `blocked` / `failed` / `cancelled`) |
| **HYP** (hypothesis) | `HYP-NNN`  | `active ŌåÆ confirmed` / `refuted` / `stalled` ŌĆö accumulates verdicts in `validation_log`            |
| **PROP** (proposal)  | `PROP-NNN` | `open ŌåÆ accepted` / `rejected` / `deferred` / `duplicate`                                          |

State-machine diagrams ŌĆö see [4]. Day-to-day workflow ŌĆö see [7] (HATS-358, not yet written).

**Plan home.** A Task's plan lives at exactly one canonical path ŌĆö
`<ai_hats_dir>/tracker/backlog/tasks/<ID>/plan.md` ŌĆö authored directly there via
Write/Edit after `task transition <ID> plan`. `.claude/plans` is **not** a plan
home: ai-hats ignores it (the import path was removed in HATS-637), so any file
there is inert plan-mode scratch, not the plan. Discipline + draftŌåÆtracker
procedure ŌĆö skill `plan-discipline`; readiness routing ŌĆö skill `plan-gate`;
per-section enforcement ŌĆö the engine gate (HATS-635).

## Attachment

A file attached to a Task via `ai-hats task attach add`. Blob lives in
`<ai_hats_dir>/tracker/backlog/tasks/<ID>/attachments/<name>`; the manifest
entry ŌĆö `name`, `digest` (12-char SHA-256 prefix), `added`, `note` ŌĆö is stored
in `task.yaml::attachments[]`. A pre-commit hook (HATS-402) refuses commits
that add or modify files under `attachments/` without a corresponding
manifest entry; the only legal path is the CLI.

## Reflect

The feedback loop that turns session evidence plus active HYP / open PROP into actionable items. CLI subcommand Ōåö spawned role:

| CLI subcommand                  | Spawned role                             | Mode                         | Purpose                                                                                                                                                                                                  |
| ------------------------------- | ---------------------------------------- | ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `ai-hats reflect session`       | `session-reviewer`                       | non-interactive              | Per-session retrospective: HYP verdicts + PROP-on-self-problem. Auto on `session_end` (policy `always` / `smart`) or on demand.                                                                          |
| `ai-hats reflect hypothesis`    | `judge-auditor` then `judge`             | autopilot + HITL (two-phase) | Bulk triage: Phase 1 (`judge-auditor`, headless, read-only) produces a draft; Phase 2 (`judge`, HITL) discusses + ack's mutations. `--headless` runs Phase 1 only (CI / cron-safe). HATS-513 / ADR-0007. |
| `ai-hats reflect all`           | `judge`                                  | interactive (HITL)           | Deprecated ŌĆö single-phase bulk triage with runtime mode-switch in the protocol skill. Kept for one bake cycle while `reflect hypothesis` rolls out; removal tracked as a follow-up task.                 |
| `ai-hats reflect role <target>` | `auditor-for-role` then `judge-for-role` | autopilot + optional HITL    | Coherence audit of a single role: autopilot pass first, then interactive review.                                                                                                                         |
| `ai-hats reflect roles`         | `judge-for-role` *                       | per-role HITL                | Bulk role audit ŌĆö spawns one session per project role.                                                                                                                                                   |
| `ai-hats reflect issue`         | (no role)                                | non-interactive              | Log a supervisor observation as a new HYP, or merge into an active one.                                                                                                                                  |

**Naming note:** Two-phase pairs are symmetric:

- Backlog triage: `judge-auditor` (L0, read-only audit) ŌåÆ `judge` (L1, HITL + ack'd mutations). Entry: `ai-hats reflect hypothesis`.
- Role coherence: `auditor-for-role` (L0, non-interactive coherence pass) ŌåÆ `judge-for-role` (L1, interactive review). Entry: `ai-hats reflect role`.

L0 baselines (`base-auditor`) forbid CLI mutations and source-file edits; L1 baselines (`base-judge`) permit ack'd CLI from a whitelist. See [`library/core/traits/base-auditor/`](../library/core/traits/base-auditor/) and [`base-judge/`](../library/core/traits/base-judge/). `hypothesis-intake` exists for Haiku-class observation classification but is **not** wired into `reflect *` directly.

Practical recipes ŌĆö see [5]. Pipeline architecture ŌĆö see [8].

## Artifacts

What ai-hats persists on disk during normal use.

- **`ai-hats.yaml`** ŌĆö project config. Fields: `schema_version`, `migration_step`, `provider`, `active_role`, `default_role`, `task_prefix`, `customizations`, `feedback`, `library_paths`, `venv_path`, `harness`. `harness` (HATS-764) pins the install-source channel ŌĆö see **Harness source / channel**. `schema_version` describes the yaml format; `migration_step` (HATS-471) is a monotonic counter for one-shot migrations replayed at bump time ŌĆö each entry runs once per project, then the gate short-circuits. Source of truth for composition. Apply changes by re-running `ai-hats self init` (idempotent) or `ai-hats self update` (which folds bump in); verify with `ai-hats config status`. Full walkthrough ŌĆö see [6].
- **Harness source / channel** ŌĆö the per-project choice of where `ai-hats self update` pulls ai-hats from, persisted as `harness.channel` in `ai-hats.yaml` (HATS-764). Three channels map to three audiences: **`local`** (ai-hats dev) ŌĆö an editable install of a working tree (`uv pip install -e <path>`), a moving target reinstalled in place with no versioned dir; **`edge`** (own repos) ŌĆö `git+https://<repo>@<branch-HEAD-sha>`, a moving target installed into `versions/<sha>/`; **`stable`** (end users, the **default**) ŌĆö `ai-hats==<latest-tag>` from PyPI, pinned and semver-monotonic, installed into `versions/<tag>/`. A pure resolver maps `channel ŌåÆ (version_id, install_spec, mutable)`: **`version_id`** names the `versions/<version_id>/` dir (a `sha` for edge, a tag for stable; `None` for local's in-place install), and **`mutable`** selects the downgrade guard ŌĆö the moving-target channels (edge/local) keep the git ahead/diverged guard, while `stable` runs a **semver-monotonic** guard (refuse a published tag whose version is lower than installed; override `--force-downgrade`). Set it with `ai-hats config set --channel {local|edge|stable}` (the wizard and config surface never hand-edit yaml); `AI_HATS_REPO_URL` overrides the edge repo. An absent `harness:` block loads as `stable` and saves byte-clean. See **Versioned install layout**.
- **SessionReview** ŌĆö `<ai_hats_dir>/sessions/retros/sessions/<id>.md`. Output of `session-reviewer`: `summary`, `observations`, `hypothesis_verdicts`, `proposal_actions`. Schema `hats-session-review/v1`. Consumed by the next reflect cycle. Detail ŌĆö see [5].
- **JudgeReport** ŌĆö `<ai_hats_dir>/sessions/retros/judge/<UTC-ts>-report.md`. Output of `ai-hats reflect hypothesis` Phase 2 (or the legacy `ai-hats reflect all`) ŌĆö HYP closures plus PROP decisions for one triage session. Detail ŌĆö see [5].
- **JudgeDraft** ŌĆö `<ai_hats_dir>/sessions/retros/judge/<UTC-ts>-draft.md`. Output of `ai-hats reflect hypothesis` Phase 1 (`judge-auditor`, headless). Carries proposed verdicts + a `## Proposed mutations` CLI checklist that Phase 2 executes after supervisor ack. With `--headless` this is the only artifact (no Phase 2). HATS-513 / ADR-0007.
- **RoleCoherenceReport** ŌĆö `<ai_hats_dir>/sessions/retros/role-coherence/<UTC-ts>-<target>.md`. Output of `ai-hats reflect role` ŌĆö findings on internal contradictions in a role composition. Detail ŌĆö see [8].
- **Versioned install layout** ŌĆö `<ai_hats_dir>/versions/<version_id>/` (a per-version venv, keyed by an edge `sha` *or* a stable tag ŌĆö HATS-764) + a `versions/current` pointer-file (HATS-647). A managed `ai-hats self update` installs the new version into a fresh `versions/<sha>/` and atomically flips `current` (tmp-write + `os.replace`) only after a successful install + verify ŌĆö so an update never mutates the venv a live run executes from, and an interrupted update never bricks the tool (`current` always points at a complete venv; on crash it stays on the old `sha`). The bash launcher resolves `current ŌåÆ versions/<sha>` once at run start and **exports `AI_HATS_VENV`**, pinning the whole run (hooks, sub-agents, bash-chained `ai-hats ŌĆ” | ai-hats ŌĆ”`) to one `sha` even if `current` flips mid-run (pin-at-spawn). Applies only to the managed default venv; editable / dev installs and explicit `AI_HATS_VENV` / yaml `venv_path` overrides bypass versioning, and a pre-versioning `.venv` is the lazy fallback. Completeness is gated by a **`.complete` sentinel** written last ŌĆö only after a fully-successful install + verify ŌĆö so it is robust to a build killed mid-pip, independent of pip's internal file-write ordering (HATS-648; HATS-790 removed the legacy `bin/ai-hats` console-script proxy this note used to contrast against ŌĆö the launcher now execs `bin/python -m ai_hats`); `read_current_sha` resolves a `sha` only when its venv carries the sentinel and a runnable `bin/python`. Incomplete residue (a `versions/<sha>/` lacking `.complete`, e.g. a crashed install) is reclaimed idempotently by the **`version_recovery`** sweep ŌĆö TTL-guarded, run at both the `create_session` chokepoint and `self update` start, never touching `current` or a complete dir (HATS-648 / R1). Complete-but-superseded versions are reclaimed by the **liveness-ref GC** (HATS-649 / R2, next entry).
- **Liveness ref / orphan-version reclaim** ŌĆö `<ai_hats_dir>/versions/.refs/<root_pid>.json` carrying `{run_id, root_pid, start_time, sha}` (HATS-649 / R2). Every run writes/refreshes its ref at the `create_session` chokepoint, recording the managed `sha` it executes from (`sys.prefix` under `versions/`; legacy `.venv` / editable installs write none). `reclaim_orphan_versions` then removes any **complete, non-`current`** version that no **live** ref pins ŌĆö **reclaim-on-certain-death**: a ref is live iff its `root_pid` is alive *and* the OS-reported `start_time` (via `ps -o lstart=`, with an `os.kill` fallback) still matches, so a reused pid is detected as dead. Single-host, **no TTL** for this pass; dead refs are swept in the same pass so refs never leak. Conservative ŌĆö a live ref, `current`, a `self update`'s `target_sha` (`keep_shas`), an incomplete dir, or the `.refs` store itself is never touched; the legacy `.venv` lives outside `versions/` and is reclaimed only by HATS-653. Recovery is an injected `EnvironmentRecovery` collaborator on `SessionManager`, so both the HITL (`WrapRunner`) and Automate (`SubAgentRunner`) paths converge through it ŌĆö closing the gap where the sweeps ran only on the WrapRunner path. The reclaim and the `self update` install both run under the **version GC/acquire lock** (HATS-650 / R3) so a concurrent run can never reclaim a version mid-flip.
- **Legacy `.venv` reclaim** ŌĆö `reclaim_legacy_venv` removes the orphaned pre-versioning `<ai_hats_dir>/.venv` once the versioned install is authoritative (HATS-653 / Phase B). After lazy migration the legacy `.venv` only ever resolves as a *fallback* (launcher / `venv_path` precedence consult it solely when `versions/current` is absent or broken); never updated again, its fallback value only decays. The single guard is **`current_run_sha(project_dir) is not None`** ŌĆö running from a complete `versions/<sha>/` venv proves in one predicate that (a) we are *not* on `.venv` (so discarding it can't pull the rug from the live interpreter), (b) no `AI_HATS_VENV` / yaml `venv_path` override and no editable / dev checkout is active (each resolves `current_run_sha ŌåÆ None`, so a user-owned venv is never touched), and (c) a healthy versioned venv exists. Removal is via `safe_delete.discard` (idempotent), the reclaimed path returned for no-silent-caps logging by the caller, run at the same two convergence points as the R2 reclaim ŌĆö the `create_session` chokepoint (`EnvironmentRecovery`) and `self update` start. The **first** migration `self update` runs *from* `.venv` (`current_run_sha ŌåÆ None`) and correctly skips; the reclaim converges on a later versioned invocation. Reclaim is **reversible**: if a versioned install later breaks and `.venv` is gone, an arbitrary command fails loud with "Run: ai-hats self update", whose launcher `heal_if_needed` recreates `.venv` and rebuilds versioned ŌĆö so B deliberately drops the *silent stale-fallback* (superseding the HATS-649 "preserve `.venv`" note) in exchange for bounded disk and never running ever-staler fallback code.
- **Version GC/acquire lock** ŌĆö `<ai_hats_dir>/versions/.gc.lock`, a crash-safe `fcntl` advisory lock (via `filelock` ŌĆö the same library + pattern as the worktree locks, a ~15-line wrapper, not a hand-rolled lock) serializing the two mutating sections of the versioned layout (HATS-650 / R3): the **acquire** section of `self update` (the `version_recovery` sweeps + install/reuse + the `current` flip) and the **GC** section at the `create_session` chokepoint (`sweep_incomplete_versions` + `reclaim_orphan_versions`). It closes the corruption window where an installer has written `.complete` but not yet flipped `current`: in that window the target is a complete, non-`current`, unreferenced dir, so a *concurrent* GC in another process would reclaim it and the flip would land on a deleted dir (bricking the tool) ŌĆö `keep_shas` only protects the installer's own process. The acquire site **blocks** with a generous timeout (a second concurrent `self update` waits, then a clean "another update in progress" exit on `VersionLockError`); the hot GC site is **opportunistic** ŌĆö a short timeout, swallowed, so a held lock skips the GC rather than ever blocking or breaking session start (the next invocation converges). Because it is an `fcntl` lock the kernel releases it on process death, so a `kill -9` while held never wedges future cleanup ŌĆö never a create-lockfile/delete-on-exit scheme. Single-host (`fcntl` is unreliable on NFS/SMB).
- **Dormant versioned-layout advisory** ŌĆö a non-mutating hint printed at the end of a managed `self update` when a stale host launcher is silently ignoring the versioned layout (HATS-655). Trigger is the runtime symptom, not a version number: a complete `versions/<sha>/` install **pre-existed** this update (captured at entry) yet the managed updater ran from the legacy `.venv` (`current_run_sha ŌåÆ None`) on a managed install (`_is_managed_install` ŌĆö excludes overrides / editable). On a managed install the only reasons to be on `.venv` despite an existing versioned install are a launcher predating `versions/current` resolution (HATS-647) or the **first** migration update; the pre-existing gate excludes the latter, leaving the stale launcher. The hint **names what is off** (crash-safe blue-green updates, orphan-version GC, legacy `.venv` reclaim), references the active `versions/<sha>`, and gives the one-time host-level fix (`curl ŌĆ” install-launcher.sh | bash`). **Non-mutation invariant:** the advisory only prints ŌĆö it NEVER writes the host launcher. The launcher is host-global (one entry point for ALL projects) and the least-recoverable component (a broken venv self-heals; a broken launcher cannot ŌĆö it *is* what runs `ai-hats`), so a per-project `self update` writing it would risk cross-project breakage and a bootstrap-brick; refreshing the launcher stays a deliberate host-level act.

## Host tool & self-location

How ai-hats is invoked and which code actually runs. The throughline: ai-hats is a **host tool driven by the launcher / `python -m ai_hats`**, never a project app-venv dependency. Full rationale: [ADR-0010](adr/0010-bootstrap-paradox-self-location-and-forward-safe-config.md).

- **Managed venv / managed-venv invariant** ŌĆö ai-hats is a **host tool**, not a project dependency. It is invoked by the host-global bash launcher at `~/.local/bin/ai-hats` (`AI_HATS_LAUNCHER_DEST`), which resolves a per-project **managed venv** and exec's `<venv>/bin/python -m ai_hats "$@"`. The invariant: ai-hats lives in a venv ai-hats *manages* ŌĆö the default `<ai_hats_dir>/.venv` (or a `versions/<sha>/` blue-green dir, see **Versioned install layout**), an explicit `AI_HATS_VENV` pin, a yaml `venv_path:` override, or an editable dev checkout ŌĆö and is driven via `python -m ai_hats`. It is **never** a dependency of a project's own application venv. HATS-790 (Alt 5) removed the `[project.scripts] ai-hats` console-script generator, so **no venv materialises a `<venv>/bin/ai-hats` proxy**; the only `bin/ai-hats` worth invoking is the HOST launcher. Venv health is probed by `bin/python` + `python -c "import ai_hats"` (importability), not by the removed binary. The classifier of sanctioned vs not is the pure `classify_invocation` (see **self-location guard**).
- **Self-location guard** ŌĆö a runtime backstop (`ai_hats.self_location.classify_invocation`, wired by `_guard_self_location` into `main_entry`) that **refuses-and-instructs** when ai-hats is running from a FOREIGN (non-managed) venv ŌĆö the residual **stray shadow** case after HATS-790 closed the console-script vector (HATS-791, Alt 3). The decision is a **pure** function returning `"sanctioned"` or `"foreign"`; on `"foreign"` it prints `remediation_text` to stderr and `sys.exit(3)`. It biases **HARD toward fail-open**: `"foreign"` is returned only when positively certain the running interpreter (`sys.prefix`) is a real venv that is none of the sanctioned shapes ŌĆö every ambiguity (no project, unresolved venv, any path error, an editable host clone, an under-`.agent/ai-hats/` managed prefix) resolves to `"sanctioned"`. **Scoped to an existing managed venv:** the guard only refuses when the venv ai-hats *would* resolve for this project actually exists on disk (nothing to shadow ŌćÆ nothing to protect). **Info commands are exempt** ŌĆö `--version`, `--help`/`-h`, `--tree`, and a bare `ai-hats` resolve no project state, so `_is_guard_exempt_invocation` skips them. Wired into `main_entry` (the real-invocation path) but **NOT** the `main` click group, so in-process `CliRunner` tests bypass it. **Operator escape hatch:** `AI_HATS_SKIP_SELF_LOCATION_GUARD=1` (`SKIP_ENV_VAR`) always returns `"sanctioned"`. It is defense-in-depth, not a primary gate: a missed shadow merely reproduces pre-guard behaviour while a false positive would brick a good CLI.
- **Stray shadow** ŌĆö a stale `ai-hats` reachable on `$PATH` ahead of the host launcher, typically a project app-venv into which a user once `pip install`ed ai-hats. Before HATS-790 such a venv carried a generated `bin/ai-hats` console script that `direnv` (or any rc prepending a venv `bin/`) would reach first, silently running stale code mis-resolved against the wrong project. A shadow is **only ever a PATH-resolution outcome** (absolute-path invocations are immune ŌĆö see **out-of-band recovery**). HATS-790 removed the generator that created the common shadow; the **self-location guard** is the runtime backstop, and a **stray-shadow detector** WARNs about any remaining one: `ai_hats.cli.maintenance.find_stray_launchers` (python) and its bash twin `detect_stray_launchers` in `bootstrap.sh` scan `$PATH` for `ai-hats` executables whose resolved path differs from the sanctioned launcher. Both **NEVER delete** (destructive-actions rule) ŌĆö they instruct the user to uninstall ai-hats from the shadow's venv or remove the file and front the host launcher on PATH.
- **Out-of-band recovery** ŌĆö the recovery path for when in-band `ai-hats self update` **cannot heal itself** (a managed venv broken badly enough ŌĆö deleted site-packages, dangling interpreter, a foreign shadow ŌĆö to be unrunnable, the *bootstrap paradox*: the tool that must do the fix is the broken thing). `scripts/bootstrap.sh` is the canonical hatch (HATS-791) and is **paradox-immune** by two properties: it is always **fetched fresh** (`curl -LsSf ŌĆ”/bootstrap.sh | bash`) and it drives the launcher by **ABSOLUTE path** (`"$LAUNCHER_DEST"`), never the on-`$PATH` `ai-hats` ŌĆö so a stray shadow cannot intercept it (an absolute path does not consult PATH: **absolute-path immunity**). The dedicated recovery entrypoint is the **`--repair`** flag: it force-reinstalls the launcher, removes only the framework-managed default venv (`.agent/ai-hats/.venv` + `versions/`, never a user-owned `AI_HATS_VENV` override), then re-runs `"$LAUNCHER_DEST" self update` to rebuild from scratch. Idempotent; also runs the stray-shadow detector. Usage: `curl -LsSf https://github.com/muratovv/ai-hats/raw/master/scripts/bootstrap.sh | bash -s -- --repair`.
- **Config forward-compat policy** ŌĆö how an OLDER ai-hats handles an `ai-hats.yaml` a NEWER ai-hats wrote, keyed off `KNOWN_SCHEMA_VERSION` (`ai_hats.models`, currently 4 ŌĆö the highest `schema_version` this binary understands). Two complementary rules (HATS-792): **(1) same-version unknown fields are preserved, not dropped** ŌĆö `ProjectConfig.from_yaml` pre-strips unknown top-level keys before `extra="forbid"` validation (so the older binary survives a field a newer one added without a schema bump, e.g. `migration_step`), but stashes the popped `{key: value}` on a `_extra` `PrivateAttr` that `to_dict` merges back, so readŌåÆwrite **round-trips** the field instead of silently deleting it on the next `save()` (mirrors the `TaskCard.extras` capture-on-load / merge-on-dump pattern); the HATS-581 stderr WARN still fires so the vanish-from-the-typed-model stays observable. **(2) a genuinely newer schema fails loud** ŌĆö when on-disk `schema_version` exceeds `KNOWN_SCHEMA_VERSION`, `from_yaml` raises `ProjectConfigError` pointing at `ai-hats self update` rather than misreading the file as v4 and risking a clobber, and a matching **`save()` clobber guard** refuses to overwrite a file whose on-disk `schema_version` is newer than this binary knows (defending bypass paths that build a config without first loading the existing file). Preserve covers *same-version* skew; fail-loud covers *newer-schema* skew; the two never overlap.

## Composition & pipeline internals

Names for the framework's composition pipeline + runtime split. Lock these in code, docs, and conversation so the "who owns the prompt" boundary stays unambiguous. Full rationale: [ADR-0005](adr/0005-composition-and-pipeline-value-contract.md).

- **CompositionResult** ŌĆö the flat, immutable output of `Composer.compose(role, overlays=...)`. `@dataclass(frozen=True)`; modifications via `with_*` methods only (`with_injection_override(text)` returns a new instance). Carries `priorities`, `rules`, `skills`, `injections`, plus provenance maps (`trait_injections`, `role_injection`, `overlay_injection`).
- **Pipeline funnel** ŌĆö the producer-emits / consumer-may-ignore convention by which pipeline steps thread state. Producer puts a key in the merge delta; consumer may or may not pick it up. **Value contract:** `obj is None` and `key not in ctx` are identical (the framework drops `None` values at the merge boundary); `""` / `0` / `False` / `[]` are valid non-absent values whose semantics differ from "absent". Never use magic `""` to signal "no value" ŌĆö emit `None` or omit the key.
- **HITL runner** ŌĆö `WrapRunner`. The human-in-the-loop runner: a user is at the keyboard, the agent runs interactively under a PTY proxy. **Has no `system_prompt_override` channel** ŌĆö prompt injection is meaningless here. The role's full composition reaches the agent through `composer.compose(...)` + `build_session_prompt` inside `run_session`.
- **Automate runner** ŌĆö `SubAgentRunner`. The automation runner: subprocess invocation with a required `task` argument. Used for sub-agent fan-out, batch / non-interactive `ai-hats execute`, and pipeline-driven spawns. **Accepts** `system_prompt_override` (HATS-267 use case ŌĆö caller-supplied prompt replaces the composed injection text).
- **Composition snapshot** ŌĆö `_composition_snapshot(assembler, role, result) -> dict`. Audit-only structural snapshot emitted into `Session.init_audit`. Lives in a separate channel from data-producing pipeline steps (ą¤4 in ADR-0005) ŌĆö a producer step does NOT piggyback composition data into its `produces` set for downstream routing.
- **Materialization facade** ŌĆö `src/ai_hats/materialize.py`. Single derivation point for "compose role X for this project" (HATS-456 / ADR-0005 Phase 2). Exposes `compose_for_role(assembler, role) -> CompositionResult` ŌĆö a thin wrapper around `assembler.composer.compose(role, overlays=assembler._get_overlays(role))`. Every runtime / pipeline consumer (HITL runner, Automate runner, `MaterializeSystemPrompt` step, `Assembler.set_role` writer, hook-install / status / bump compose-only sites) routes through this function. Direct `composer.compose(..., overlays=...)` outside the facade is a drift signal ŌĆö pinned by `tests/test_no_direct_compose_outside_facade.py`. The build surface (`build_session_prompt`, `build_system_prompt`, `_build_meta_prompt`) stays runtime-specific per ą¤2 in ADR-0005.

## Session-end output blocks

## Canonical base branch

The name (or names) of the branch that worktrees are expected to be created from and merged back into. Hardcoded to `master` and `main`, in that priority order (HATS-518); the first one that actually exists in the repo is the comparison target.

- **Why it matters.** `WorktreeManager.create()` captures whatever branch the main repo's HEAD currently points at as the worktree's `_original_branch` ŌĆö and that is the branch `ai-hats wt merge` later lands commits on. Two silent-wrong-branch failure modes:
  - *Create-time.* Operator parks the main repo on a feature branch before `wt create` / `task transition <ID> execute`. The worktree quietly inherits that branch as its merge target; CLI reports "merged" while master never sees the work.
  - *Merge-time.* Even when create-time HEAD was on a canonical base, the main-repo HEAD can wander off `_original_branch` between create and merge ŌĆö manual `git checkout`, IDE branch-switch, a peer agent operating directly in the main repo without a linked worktree. `_fast_forward_merge` / `_squash_merge` run `git merge` in the main-repo cwd, so the merge lands on whatever branch is currently checked out ŌĆö not on `_original_branch`.
- **Create-time guard.** `assert_head_is_canonical_base()` in `ai_hats.worktree` refuses both `wt create` and `task transition execute` when HEAD is not on a canonical base (HATS-518) ŌĆö raises `WorktreeBaseBranchError`. Detached HEAD, non-git directories, and exotic repos that have neither `master` nor `main` are passed through (no canon to compare against).
- **Merge-time guard.** `WorktreeManager.merge()` refuses when `git rev-parse --abbrev-ref HEAD != self._original_branch` (HATS-533) ŌĆö raises `WorktreeBaseBranchMismatchError` BEFORE any mutation. Positioned ahead of `_check_clean` / `_check_drift`: with HEAD wrong, drift is asking the wrong question. No-op for legacy states where `_original_branch is None`.
- **Recovery (both guards).** `git checkout <expected>` in the main repo, then retry. No work lost in either case, but the surfaces differ:
  - *Create-time refusal* (HATS-518) ŌĆö no worktree exists yet; the refusal aborts before `git worktree add` runs. Retry creates the worktree fresh once HEAD is on a canonical base.
  - *Merge-time refusal* (HATS-533) ŌĆö the worktree dir and worktree branch are preserved untouched; the refusal happens before `_check_clean` / `_check_drift` / the actual `git merge`. Retry from the corrected HEAD finishes the merge as if the refusal hadn't happened.
    Both CLI surfaces (direct `wt merge` and `task transition done`) emit a copy-pasteable recipe.
- **`--force` / `--accept-drift` do NOT bypass either guard.** `--force` is the dirty-worktree consent; `--accept-drift` is the moved-base consent. Neither addresses wrong-branch protection ŌĆö three independent safety contracts, three independent flags.
- **Not configurable.** Hardcoded two-name list, no override flag. If a project needs a different base, raise a ticket with the second use case ŌĆö until then, KISS / design-minimalism.

## Worktree data-transfer

The mechanism by which **gitignored** data crosses the `ai-hats wt` boundary in either direction ŌĆö **carry-in** (seeded *into* a worktree at `wt create`) or **carry-out** (harvested *out of* a worktree before `wt merge` / `wt discard` teardown). Designed in ADR-0012 (HATS-821); gates HATS-775 (carry-in) + HATS-818 (carry-out).

- **Why it matters.** `WorktreeManager.create()` populates a worktree only from the base branch's *tracked* git state (`git worktree add`, `src/ai_hats/worktree.py`), so anything gitignored is, by default, absent on entry **and** destroyed on teardown. Confirmed loss: a supervisor's `.hunk/notes.json` review notes were silently deleted by `wt merge` (HATS-818) ŌĆö `_check_clean` guards only *tracked* changes (`git status --porcelain`), so the gitignored sidecar is invisible to it.
- **Hooks-first, tool-agnostic.** The primitive is a **worktree lifecycle hook** ŌĆö a component-declared script run at `wt_in` (after create) or `wt_out` (before every teardown route), declared in `SKILL.md` frontmatter `ai_hats:` and collected/materialized exactly like `runtime_hooks` (`RuntimeHook`, HATS-597). ai-hats core hardcodes no tool-specific path: the `hunk-review-comments` skill ships a `wt_out` hook that runs the existing `hunk-notes.sh consume` (DOTS-157). The common path operations are sugar over a **built-in `capture` hook**: `seed_in: [{path, mode}]` and `harvest_out: [{path, on, guard}]`.
- **Resolution kinds.** seed (`wt_in`): **copy** by default (a per-worktree copy keeps worktrees isolated) or **reference** (read-only shared, for "I need the same file not a fork" ŌĆö the tracker is served by the HATS-524 redirect, not a seed); write-through to a *mutable* shared dir (`.venv`/`node_modules`) is rejected (concurrent-worktree race). harvest (`wt_out`): **backup+guard** built-in (back up unconditionally, then refuse a non-empty teardown until drained or `--accept-harvest-loss`), or a **custom hook** (fail-closed: a non-zero exit aborts teardown). Two kinds are *not* mechanism cases: **redirect-or-none** (shared mutable state read in place via HATS-524) and **doc/convention-fix** (e.g. the editable-venv test trap, HATS-641).
- **Security.** Arbitrary scripts run at lifecycle boundaries ŌåÆ same trust model as `runtime_hooks` (composed-only, materialized, logged, not sandboxed). Built-in seeds only gitignored paths, re-checked at merge (an agent can `git add -f`); credential files (HATS-776) are excluded from built-in harvest backups; built-in backups land gitignored under `.agent/` keyed by `_state_key`, never `/tmp` or a tracked path.

## Git-hook surface

The `.githooks/` tree ŌĆö **generated by composition, not tracked in git** ŌĆö and the machinery that keeps it consistent (HATS-088, HATS-593).

- **Dispatcher** ŌĆö `.githooks/<event>` written from `src/ai_hats/templates/githooks/dispatcher.sh` (marker-guarded; leaves a foreign hook of the same name alone). Event-agnostic: resolves its event from `basename "$0"`, exports it as `AI_HATS_HOOK_EVENT`, then runs `<event>.d/*` lexicographically ŌĆö the first non-zero exit aborts the chain. **Fail-closed backstop (HATS-593):** before running, it reads the manifest and stats each expected managed `<event>.d/*` entry; if one is missing or non-executable it exits 1 ("hooks corrupt ŌĆö run `ai-hats self init`") rather than silently running a degraded gate. An event with no managed entries in the manifest is never blocked.
- **`.ai-hats-manifest`** ŌĆö `.githooks/.ai-hats-manifest`, a newline list of every ai-hats-managed relpath (`<event>.d/<skill>-<basename>` plus each `<event>` dispatcher). Drives stale-removal on re-install and the dispatcher's fail-closed stat check.
- **Managed-hook drift net** ŌĆö the session-start check (`Assembler.sync_hooks`, HATS-833) that re-materializes any drifted managed-hook surface (git, provider runtime, worktree) so a launch never runs against stale hooks. Drift-gated and fail-open. Replaced the former `ai-hats self sync-hooks` command + post-merge/post-checkout git-event self-heal (both removed) ŌĆö healing is session-start only.

## Safe-delete trash bin

Single point of truth for destructive filesystem ops in ai-hats core (HATS-470). Replaces the historical pattern of raw `path.unlink()` / `shutil.rmtree()` / in-place `path.write_text(new)` calls.

- **`ai_hats.safe_delete`** ŌĆö module exposing `discard(path, *, reason, project_dir)`, `replace(path, new_content, *, reason, project_dir)`, `session_summary()`, `session_root()`, `reset_session()`, plus `TrashFullError`. `discard` moves a file/dir/symlink to the current trash session; `replace` snapshots the old content to trash, then atomic-writes the new bytes. Symlinks are unlinked (link only ŌĆö target preserved, original target written to a sidecar `.symlink` file).
- **Trash session directory** ŌĆö `${TMPDIR:-/tmp}/ai-hats/trash-<utc-ts>-<pid>-XXXXXX/`. One per process, lazily created on the first destructive op; `tempfile.mkdtemp` guarantees uniqueness across concurrent ai-hats invocations. Contents mirror the project-relative path of each victim, plus a `MANIFEST.md` recording every op (timestamp, kind, reason, original ŌåÆ trash path). External-to-project paths land under `_external/<abs-tail>/`. Recovery: `cp -r <session>/<rel> <project>/<rel>`. **Not auto-cleaned** ŌĆö relies on OS `/tmp` retention.
- **`AI_HATS_TRASH_DIR`** ŌĆö env var. Overrides the trash base directory (the `trash-<ts>-<pid>-XXXXXX/` subdir is still created underneath). Special sentinel `AI_HATS_TRASH_DIR=-` enables **hard-delete mode**: no snapshots, one WARN to stderr per op, no session directory. Intended for CI / ephemeral environments where snapshot value is zero.
- **`TrashFullError`** ŌĆö `OSError` subclass raised when the snapshot can't land (ENOSPC, read-only filesystem, missing permission). Callers (bump / init) treat it as fatal ŌĆö partial migrations without a recoverable snapshot violate the trash-bin contract.
- **Inline whitelist marker** ŌĆö `# safe-delete: ok <reason>` on the same line as a raw `unlink` / `rmtree` / `rmdir`. Use only for genuinely safe cases (empty-dir cleanup, internal `.tmp` from atomic-write, session-cache rebuild, framework-managed manifest). Enforced by the `pre-commit-no-raw-destructive.sh` hook in the `git-mastery` skill ŌĆö refuses commits that introduce raw destructive ops without either being in `src/ai_hats/safe_delete.py` or carrying the marker. Override: `AI_HATS_NO_RAW_DESTRUCTIVE_SKIP=1 git commit ...`.

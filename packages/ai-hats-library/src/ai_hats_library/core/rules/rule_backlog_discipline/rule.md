# Rule: Backlog Discipline

Applies to all three backlog item types — **tasks** (`HATS-NNN`), **hypotheses** (`HYP-NNN`), and **proposals** (`PROP-NNN`).

1. **CLI-only.** All backlog operations via the `rack` CLI — lifecycle / reads / links (`rack transition` / `rack create` / `rack context` / `rack ls`), field edits (`rack transition --set`), documents (`rack transition --attach`/`--rm`/`--freeze`), and the `rack hyp …` / `rack proposal …` groups. Never read or edit `<ai_hats_dir>/tracker/backlog/**` directly. This covers the whole `tasks/<ID>/**` subtree — task.yaml AND its documents. Direct `mkdir`/`mv`/`echo > task.yaml`/`sed -i` under the backlog are violations. **Carve-out for `plan.md`:** `tasks/<ID>/plan.md` is editable directly via Write/Edit — it is the role's primary plan-stage deliverable, not a state mutation (parallel to the `base-judge §L0 carve-out` for analyst reports). The scaffold is created by `rack transition <ID> plan`; the content is the agent's responsibility. It is authored straight into the tracker — never via `.claude/plans`; the canonical flow is skill **plan-discipline**.
2. **Work log cadence.** Log after every significant action on a task: approach changes, file deletions, branch operations, milestone completions. For HYP/PROP, append to `validation_log` / `votes` via CLI.
3. **State transitions immediate.** Transition state when work changes phase — no stale states. Applies to task lifecycle (`brainstorm → … → done`), HYP status (`active → confirmed | refuted | stalled`), and PROP status (`open → accepted | rejected | deferred | duplicate`).
4. **Completion gate & review stop.** A task is moved to `review` when execution and documentation are finished. The executing agent MUST stop at `review` and await reviewer/user approval — agents MUST NOT self-advance from `review` to `done`. The reviewer drives `review → done`.
5. **HYP vs technical risk.** HYP / PROP backlog is reserved for **agent-behaviour hypotheses** (how the agent makes decisions, which triggers fire, which practices yield results). **Do NOT file** HYP / PROP for carry-over technical / migration / harness / documentation / coordination risks surfaced in review — those go to `rack transition <FOLLOW_UP_ID> --log "…"` (or as plan-stage additions on the relevant task), not to a new HYP "just in case". Cost rationale: HYP / PROP carry expensive scaffolding (validation_log, votes, judge triage in reflect-all). Concrete technical risks bound to a known task live there. If a risk isn't bound to a concrete task and isn't behavioural either — memory or skip; "we'll see it when we get there" is acceptable.
6. **Premise retraction is a go/no-go event, not a doc edit.** Marking a task's strategic driver — or its sole motivating consumer — as dead while the task sits in `plan` / `execute` **is** a transition trigger. The trigger is the act of writing any of these into the card: a `[RETRACTED]` / `[SUPERSEDED]` note, *"no longer holds"* / *"driver changed"*, or removing the justification a plan rests on. **Name that edit as the trigger** — do not file it as a routine description or work-log update. Transition the task back to `brainstorm` so the value gate (`plan-gate` → `devils-advocate`) re-fires against the new premise set **before** any execute work; do not leave it in `plan` looking "ready to implement". The rationalization *"substance unchanged"* is the exact signal this rule overrides — a collapsed premise changes go/no-go even when the spec text is untouched. (Origin: HATS-795 — a retraction filed as *"Description updated… Substance unchanged"* left the task in `plan`; it reached a green ~920-LoC build before cancellation.)

## Scope

§1 applies to every role that touches the backlog (filing, lifecycle, or read).

§§2–4 and §6 apply only to roles that **own a backlog item's lifecycle** — the fix author / lead / primary agent. Roles whose protocol skill whitelists only `rack create` (e.g. L1 analyst roles like `judge-for-role`) follow §1 only; they never enter the lifecycle-owner obligations because they never own a lifecycle.

For CLI commands, lifecycle details, and plan-flow procedures → skill **hatrack** (or **backlog-manager** for a classic session).
